# Provenance Guard

A backend API for classifying whether submitted text content was written by a human or generated by AI. Built for creative sharing platforms to surface attribution transparency to readers and give creators a path to appeal misclassifications.

---

## Architecture Overview

A submitted piece of text passes through two independent detection signals — an LLM semantic classifier and a stylometric heuristic analyzer. Their outputs are combined into a single weighted confidence score, which drives both the transparency label shown to the user and a structured audit log entry. If a creator contests the result, the appeal endpoint updates the content's status and logs their reasoning for human review.

**Submission flow:**

```
POST /submit → LLM Signal → Stylometric Signal → Confidence Score → Transparency Label → Audit Log → JSON Response
```

**Appeal flow:**

```
POST /appeal → Lookup content_id → Update status to "under_review" → Append reasoning to log → JSON Response
```

**Endpoints:**

| Endpoint  | Method | Description                             |
| --------- | ------ | --------------------------------------- |
| `/submit` | POST   | Submit content for attribution analysis |
| `/appeal` | POST   | Contest a classification decision       |
| `/log`    | GET    | Retrieve recent audit log entries       |

## Detection Signals

### Signal 1: LLM Classification (Groq llama-3.3-70b-versatile)

**What it measures:** Semantic and stylistic coherence holistically — whether the text "reads" like AI output. Captures unnaturally smooth transitions, generic hedging language ("it is important to note that"), overuse of parallel structure, and lack of personal specificity.

**Output:** A float 0.0–1.0 (1.0 = high confidence AI-generated), extracted from a structured JSON response.

**What it misses:** Skilled human writers producing formal or academic prose may score high. Lightly edited AI output that has personal details added will confuse this signal.

### Signal 2: Stylometric Heuristics (pure Python)

**What it measures:** Statistical surface properties that differ between human and AI writing. Three metrics:

- **Sentence length variance** — AI text has low variance; human writing is irregular
- **Type-token ratio (TTR)** — vocabulary diversity; AI text reuses common words more
- **Informal punctuation density** — dashes, ellipses, exclamation points signal human voice

**Output:** A float 0.0–1.0 (1.0 = AI-like uniformity), computed in pure Python with no external libraries.

**What it misses:** Short texts (<50 words) don't have enough data for meaningful variance calculations. Academic or legal writing by humans naturally has low variance.

## Confidence Scoring

**Combination formula:**

```
confidence = (0.6 × llm_score) + (0.4 × stylometric_score)
```

The LLM signal gets higher weight because it captures semantic properties that stylometrics miss. Stylometrics serve as a structural check.

**Score thresholds:**

| Score Range | Attribution    | Label Shown                      |
| ----------- | -------------- | -------------------------------- |
| 0.75 – 1.00 | `likely_ai`    | ⚠️ AI-Generated Content Detected |
| 0.40 – 0.74 | `uncertain`    | 🔍 Authorship Uncertain          |
| 0.00 – 0.39 | `likely_human` | ✅ Likely Human-Written          |

The uncertain band is deliberately wide because false positives — labeling a human's work as AI — are worse than false negatives on a creative platform.

**Example submissions with different confidence scores:**

| Text (truncated)                                                              | LLM Score | Stylometric Score | Combined Confidence | Attribution  |
| ----------------------------------------------------------------------------- | --------- | ----------------- | ------------------- | ------------ |
| "Artificial intelligence represents a transformative paradigm shift..."       | 0.75      | 0.51              | 0.66                | uncertain    |
| "ok so i finally tried that new ramen place downtown and honestly?..."        | 0.10      | 0.34              | 0.20                | likely_human |
| "In conclusion, it is essential to recognize that effective communication..." | 0.90      | 0.69              | 0.82                | likely_ai    |

Scores vary meaningfully across inputs — a 0.20 and a 0.82 produce completely different labels and different user experiences.

## Transparency Label

All three label variants — exact text returned by the API and displayed to users:

**High-confidence AI (confidence ≥ 0.75):**

```
⚠️ AI-Generated Content Detected
Our system is highly confident this content was generated by an AI tool
rather than written by a human. Confidence: {pct}%.
If you are the creator and believe this is incorrect, you may file an appeal.
```

**Uncertain (confidence 0.40–0.74):**

```
🔍 Authorship Uncertain
Our system could not confidently determine whether this content was
human-written or AI-generated. Confidence: {pct}%.
This content has been flagged for transparency. The creator may file an
appeal to provide additional context.
```

**High-confidence human (confidence < 0.40):**

```
✅ Likely Human-Written
Our system believes this content was written by a human creator.
Confidence: {pct}%.
Attribution signals indicate original human authorship.
```

## Rate Limiting

Applied to `POST /submit`: **10 requests per minute, 100 requests per day per IP.**

**Reasoning:** A legitimate creator submitting their own work rarely does so more than a few times per hour. 10/minute allows normal usage (testing, resubmitting after edits) while blocking automated scripts that flood the system. 100/day is generous for a single creator but stops large-scale scraping or adversarial probing. When the limit is exceeded, the server returns `429 Too Many Requests`.

## Appeals Workflow

Creators can contest a classification by submitting a `POST /appeal` with their `content_id` and a written explanation.

**What happens:**

1. The system looks up the `content_id` in the audit log
2. Updates `status` from `"classified"` to `"under_review"`
3. Appends `appeal_reasoning` and `appeal_timestamp` to the log entry
4. Returns confirmation to the creator

No automated re-classification occurs — appeals go to a human review queue.

**Example appeal request:**

```json
{
  "content_id": "7070610a-a547-44c0-ae2d-770dea2e7f97",
  "creator_reasoning": "I wrote this myself. I am a non-native English speaker and my writing style tends to be formal."
}
```

**Example response:**

```json
{
  "appeal_received": true,
  "content_id": "7070610a-a547-44c0-ae2d-770dea2e7f97",
  "status": "under_review",
  "message": "Your appeal has been received and will be reviewed by a human moderator."
}
```

## Known Limitations

**Formal human writing scores high:** Academic abstracts, legal briefs, and professional reports written by humans have low sentence length variance and formal vocabulary — the same surface properties the stylometric signal associates with AI. A PhD student submitting their own abstract could be labeled `uncertain` or `likely_ai`. This is a known false positive risk and is the primary reason the appeals workflow exists.

**Short texts are unreliable:** Texts under ~50 words don't provide enough data for meaningful stylometric variance calculations. The system falls back to 0.5 for the stylometric signal in these cases, leaning heavily on the LLM signal alone.

## Spec Reflection

The spec helped most with the false positive framing — the hint that "a false positive is worse than a false negative on a writing platform" directly shaped the decision to make the uncertain band wide (0.40–0.74) rather than splitting at 0.5. That asymmetry is baked into the threshold design.

One divergence: the spec suggests SQLite for the audit log. I used a flat JSON file instead because it's simpler to inspect and commit for grading purposes, and the project doesn't require querying across entries in complex ways. For a real deployment, SQLite or a proper database would be necessary.
